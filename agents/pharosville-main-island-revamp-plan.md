# PharosVille Main Island Revamp Implementation Plan

Date: 2026-04-30
Status: Draft reviewed by specialized subagents
Scope: Standalone `pharosville` repository only
Primary request: Preserve current sea zones, ships, and associated logic/visuals while completely revamping the main island, lighthouse, and harbors; shrink the main island by roughly 33%; use PixelLab sprite/map generation.

## Objective

Replace the weak current main-island presentation with a cohesive, compelling,
old-school isometric RPG maritime island while preserving the current sea and
ship systems.

This is a coordinated map, asset, and renderer change. It must not become a
ship, risk-water, API, or mobile-runtime change.

## Non-Negotiable Preservation Rules

- Preserve the authored `56 x 56` map size.
- Preserve current DEWS/risk-water semantics and labels:
  - Calm Anchorage remains the large left-edge basin.
  - Watch Breakwater remains the wide top-edge band.
  - Alert Channel, Warning Shoals, and Danger Strait remain eastern-corner rings.
  - Ledger Mooring remains the only non-DEWS named risk-water area.
- Preserve ship class, scale, sprite IDs, logo placement, motion cadence, risk
  placement, route sampling, docking semantics, reduced-motion behavior, and
  hit-target behavior.
- Preserve same-origin `/api/*` access and do not touch Pages Function/API
  allowlists.
- Preserve the desktop gate: below `1280 x 760`, do not mount the world, fetch
  world data, fetch the manifest, create canvas, or decode sprites.
- Do not add runtime assets beyond the manifest budget unless explicitly
  justified. At plan time, the manifest reports 34 runtime assets and 24
  first-render assets, while the validator enforces a 34-asset cap, so this
  plan assumes in-place replacement of existing island/lighthouse/dock assets.
- Do not commit `dist/`, `test-results/`, `.env*`, PixelLab scratch outputs, or
  remote generated URLs.

  `src/systems/world-layout.ts`.
- Current lighthouse tile is `LIGHTHOUSE_TILE = { x: 18, y: 28 }`.
- Current civic center is `CIVIC_CORE_CENTER = { x: 31, y: 31 }`.
- Current island model overlay is `overlay.central-island` at
  `public/pharosville/assets/overlays/central-island.png`.
- Current overlay drawing uses hard-coded renderer placement:
  `CENTRAL_ISLAND_MODEL_TILE = { x: 31.5, y: 41.3 }` and
  `CENTRAL_ISLAND_MODEL_SCALE = 1.45` in `src/renderer/world-canvas.ts`.
- Current lighthouse rendering is not fully manifest-native. It uses
  `LIGHTHOUSE_SPRITE_CROP`, `lighthouseRenderState()`, and
  `drawLighthouseAssetCrop()` in `src/renderer/world-canvas.ts`; hit testing
  also falls back to hard-coded lighthouse target geometry in
  `src/renderer/geometry.ts`.
- Current dock placement is hard-pinned in `EVM_BAY_DOCK_TILES`,
  `OUTER_HARBOR_DOCK_TILES`, and `PREFERRED_DOCK_TILES` in
  `src/systems/world-layout.ts`.
- Ethereum harbor is special-cased:
  - background body drawing in `drawBackgroundedHarborDocks()`
  - demoted hit priority in `visualPriorityForHitTarget()`
- Current route target has been `78-82%` water by tile count. A literal 33%
  reduction of current main-island land area is expected to push water ratio to
  roughly `85%`, so tests and docs must intentionally adopt the new target.

## Desired Visual Direction

The revamped island should read as a compact Pharos maritime observatory city:

- Smaller, denser, authored central landmass with more visible surrounding sea.
- Pale limestone cliffs and seawalls.
- Terracotta roof accents.
- Dark timber piers and quay decks.
- Oxidized bronze beacon and lantern hardware.
- Warm lighthouse and harbor lights.
- Cool teal water-bounce on lower edges.
- Crisp pixel edges, low top-down isometric view, old-school 16-bit RPG style.
- Dark contact shadows under every generated sprite.

Avoid:

- Generic fantasy village.
- ClaudeVille lore or entities.
- Text, logos, chain names, UI panels, status badges, or baked analytics colors
  in generated sprites.
- New visual meaning for bridge volume, transfer activity, issuer operations,
  DEX liquidity, redemption backstops, or blacklist activity.

## High-Risk Areas From Subagent Review

1. Shrinking the main island is not a radius tweak.
   Update `mainIslandValue()`, dock constants, lighthouse mountain/clearance,
   renderer overlay placement, tests, docs, and snapshots together.

2. Water-ratio tests will fail by design.
   If the product accepts a 33% land shrink, update the documented target from
   `78-82%` to the measured new target, likely around `84-86%`.

3. Dock tiles will drift.
   Re-author every preferred dock tile so it remains a land/coast tile with
   cardinal water access after the island shrink.

4. Lighthouse rendering can drift from hitbox and beam.
   A regenerated lighthouse requires synchronized updates to manifest geometry,
   hard-coded crop constants or renderer logic, beacon point, hit tests, and
   visual snapshots.

5. Central island overlay is visually separate from map logic.
   The new tile mask and generated overlay must agree in footprint, scale,
   harbor locations, and lighthouse socket.

6. Island periphery affects DEWS water.
   `isWithinIslandPeriphery()` uses the main-island land mask. Shrinking land
   will expose additional semantic water unless the generic neutral halo is
   intentionally recalibrated.

7. Manifest count budget is already maxed.
   Replace `overlay.central-island`, `landmark.lighthouse`, and selected dock
   sprites in place. Do not add new runtime IDs in the first implementation.
   Also watch critical-byte weight and decoded-pixel cost; passing the count
   cap and PNG-dimension validator is not enough if first render becomes heavy.

8. Ethereum harbor has special draw/hit handling.
   Keep the current special case unless a broader renderer cleanup is
   intentionally tested.

## Implementation Strategy

Use a three-track implementation with a single integration pass:

1. Re-author map geometry and dock coordinates.
2. Generate/promote PixelLab assets in place.
3. Align renderer geometry, tests, docs, and snapshots.

Do not modify ship logic, ship sprites, risk-placement logic, motion rules, API
code, or desktop gating unless a test proves an incidental adjustment is needed.
