Author: claude (opus 4.7) — synthesized from four parallel domain audits at HEAD `e717b59`

## Why NFS4

NFS1, NFS2, and NFS3 shipped substantial wins: time-delta clamp, lucide slim, structural world cache + memo, static-layer cache (terrain/scene/water-overlay split), shoreline candidate cache + viewport cull, hit-test spatial index, binary-search water-path sampling, edge caching for `/api/*`, query subscription narrowing, first-settle quorum gate, vendor manualChunks + desktop modulepreload, asset/image/bundle budget validators.

The 2026-05-02 micro-pass shipped one more focused win in `asset-manager.getLoadStats`.

This plan catalogs the **remaining** opportunities found across four parallel audits (bundle/cold-load, renderer hot-path, React/data, motion/RAF). Items the audits confirmed already shipped are listed in **§ Confirmed DONE** at the bottom.

## Phase 1 — high-leverage, low-risk wins

### 1. Strip Zod schema side-effects from the desktop chunk

**Anchor:** `shared/types/index.ts` re-exports `./api-meta`, `./market`, `./report-cards`, `./pharosville`, etc., and **every one of those modules executes `z.object(...)` at top level** (e.g., `shared/types/api-meta.ts:3-19`, `shared/types/pharosville.ts:24`). Most `src/**` files import these as types only, but the side-effects run at import time because there is no `"sideEffects": false` boundary on the shared subpath.

**Symptom:** the desktop chunk contains ~58 KB raw / ~15-20 KB gzip of Zod runtime classes (`ZodObject`, `ZodString`, `ZodArray`, `safeParse`) — verified by raw-chunk grep. The `pharosville-api-client-contract.ts` zod-free split (NFS3-01) hides the contract path but leaves the `shared/types` schemas in the graph.

**Change:** split `shared/types/index.ts` into `index-types.ts` (pure re-exports of `type` only — no value imports) and `index-schemas.ts` (everything that runs `z.object`). Audit the 19 src usages — most are already `import type` and only need the type re-export hub. Mark `shared/types/*` schema files `"sideEffects": false` selectively, OR move every top-level `z.object` block into a sibling `*-schemas.ts` that production never imports.

**Risk:** any src file that *runtime-imports* a Zod value from `shared/types/*` breaks. Grep `from "@shared/types"` for non-`type` imports first. Validation chain in dev/test (which uses the schemas) must continue to import from the schema sibling.

**Validation:** `npm run typecheck` + `npm run build` + `npm run check:bundle-size`. Target: desktop chunk gzip drops by **15-20 KB**.

### 2. Hit-target snapshot: camera-only re-projection on drag/zoom

**Anchor:** `src/pharosville-world.tsx:159-162` calls `recomputeHitTargets()` on every `canvas.camera` change. Drag pan is RAF-batched (good), but each batched `setCamera` still triggers a full `createHitTargetSnapshot` (`src/renderer/hit-testing.ts:73`): rebuilds `recordsById` Map, sorts, rebuilds the spatial index over **all docks/ships/areas/graves**.

**Symptom:** ~60 full snapshots per second during a drag pan, despite zero entity additions/removals.

**Change:** add a `recomputeHitTargetsForCameraOnly(snapshot, camera, canvasSize)` path that reuses the existing record list and re-projects only the screen rects (and re-bins them in the spatial index). Wire it into the camera-change effect; keep the full rebuild on `world` swap, asset-load tick, fullscreen toggle, and selection delta.

**Risk:** correctness during drag — selections/hovers must keep targeting the right entity. Keep a fallback full rebuild on drag-end and zoom commit, and add a unit test that asserts both paths produce identical record-id ordering and identical rects when fed the same inputs.

**Validation:** `npm test -- src/renderer/hit-testing.test.ts`; Playwright drag-pan scenario + manual flame chart inspection.

### 3. Cap critical asset loader concurrency

**Anchor:** `src/renderer/asset-manager.ts:131-135` calls `loadAssetGroup(critical, manifest, signal)` with no `concurrency` option; `loadAssetGroup` defaults to `options.concurrency ?? assets.length` (line 220). With 27 critical assets, that's 27 simultaneous PNG fetches+decodes contending with the desktop JS chunk.

**Change:** introduce `PHAROSVILLE_CRITICAL_ASSET_CONCURRENCY = 6` (HTTP/2 stream-default friendly) and pass it from `loadCritical`. Keep deferred at its current cap.

**Risk:** must not regress `criticalAssetsLoaded` time on a fast connection. Measure before/after on Fast 4G in DevTools; if median TTFP regresses, raise to 8.

**Validation:** `npm test -- src/renderer/asset-manager.test.ts`; manual cold-load measure with throttle.

### 4. Trim server-only `cron-jobs.ts` from the client bundle

**Anchor:** `shared/lib/pharosville-endpoint-registry.ts:4` imports `CRON_INTERVALS` from `cron-jobs.ts` (513 lines, ~28 cron-job definitions). The desktop chunk only needs intervals for ~6 keys; the rest are server-only jobs (`sync-mint-burn`, `dispatch-telegram-alerts`, `prune-cron-history`, etc.). Confirmed at runtime: chunk contains string literals like `"compute-dews"`, `"sync-mint-burn"`, `"dispatch-telegram-alerts"`.

**Change:** create a `shared/lib/cron-intervals-client.ts` exporting only the client-relevant interval map; switch `pharosville-endpoint-registry.ts` to import from there. Keep `cron-jobs.ts` as the single source of truth — derive the client map from it at build time, or assert equality in a unit test.

**Risk:** drift between client and server cron interval values. Mitigate with the equality test.

**Validation:** `npm run check:bundle-size`. Target: ~5-10 KB raw drop, plus simpler tree-shaking that compounds with #1.

### 5. Cache lantern radial gradient (overlay ships at night)

**Anchor:** `src/renderer/layers/ships.ts:963-979`. `drawShipOverlay` calls `createRadialGradient` once per overlay-ship per frame whenever `nightFactor > 0`, against a varying mast (x,y).

**Symptom:** with ~20 overlay ships at night that's 20 gradients/frame plus ~80 `addColorStop` calls — peak GC source on the night ambient path.

**Change:** bake one normalized lantern gradient into a small offscreen canvas at module load (or per-zoom bucket) and `drawImage` it scaled — same trick as `lampLightConeSpriteCache` in scenery (`src/renderer/layers/scenery.ts:104-159`).

**Risk:** very low — gradient is purely positional.

**Validation:** night-mode visual snapshot unchanged; profiler shows zero `createRadialGradient` calls in the ship pass.

---

### 6. Convert `RenderFrameCache` to WeakMap-keyed entity lookups

**Anchor:** `src/renderer/frame-cache.ts:10-53`. `createRenderFrameCache` allocates two new `Map`s per frame; `entityFrameKey` builds `${entity.kind}:${entity.id}` strings on every `geometryForEntity` lookup. With ~50 entities and ~3 lookups each = **~150 string allocs per frame**.

**Change:** key by entity reference via `WeakMap<Entity, FrameGeometry>` (entities are stable across the world's lifetime). Eliminates both Map allocation and string keys.

**Risk:** must verify entities are referentially stable in the world payload between frames — they are because the world reference is structurally cached (NFS3-02 via `completeWorldRef`).

**Validation:** `npm test -- src/renderer/frame-cache.test.ts`; visual suite unchanged.

---

### 7. Hoist `dockRenderStates` / `graveRenderStates` to module scratch

**Anchor:** `src/renderer/world-canvas.ts:455-456`. `createWorldCanvasFrame` always builds `dockRenderStates: new Map()` and `graveRenderStates: new Map()`, mirroring the `shipRenderStatesScratch`/`flagshipByIdScratch` scratch already in place at lines 90-93.

**Change:** promote to module-level scratch + `clear()` on frame entry.

**Risk:** verify no test path retains a reference across frames.

**Validation:** `npm test -- src/renderer`; visual suite unchanged.

## Phase 2 — coherence and per-frame churn

### 8. Cache `drawLamp` halo as sprite

**Anchor:** `src/renderer/layers/ambient.ts:178-184, 232-261`. 12 village lamps × per-frame radial gradient + 3 `addColorStop` each = **36 colour-stop calls/frame** for the same logical halo. Halo varies only with `glow` (sin of phase).

**Change:** pre-bake the halo into a sprite (same `lampLightConeSpriteCache` pattern as scenery harbor lamps), modulate via `globalAlpha`. Cache becomes shared across both layer call sites.

**Risk:** very low.

**Validation:** visual snapshot of village + harbor lamp lit areas unchanged.

---

### 9. Cache sun/moon glow gradients per quantized phase

**Anchor:** `src/renderer/layers/sky.ts:260, 292`. Two `createRadialGradient` calls per frame whenever the mood is non-night/non-day.

**Change:** cache by `(width, height, zoom-bucket, mood, quantized-phase)` — mirror the `nightGradientCache` pattern in lighthouse (`src/renderer/layers/lighthouse.ts:121, 731-780`). Quantize phase to minute granularity.

**Risk:** very low.

**Validation:** dawn/dusk visual snapshots unchanged.

---

### 10. Pre-bake lighthouse beam gradient

**Anchor:** `src/renderer/layers/lighthouse.ts:669-674`. `drawLighthouseNightHighlights` builds a fresh `createLinearGradient` for each beam arm per frame because endpoints follow `sweepAngle`. Colour stops are constant; only endpoints rotate.

**Change:** either swap to a radial gradient anchored at firePoint, or pre-bake a beam sprite drawn with a rotation transform. Skips both `createLinearGradient` calls per frame.

**Risk:** small — beam edge softness must match. Visual snapshot review on night frames.

**Validation:** `npm test -- src/renderer/layers/lighthouse-night.test.ts`; visual diff at night.

### 11. Memoize `effectShipIds` on `(basePlan, selectedShipId)`

**Anchor:** `src/systems/motion-planning.ts:139` rebuilds `new Set(basePlan.baseEffectShipIds)` and conditionally `.add(selectedShipId)` on every `buildMotionPlan` call. `pharosville-world.tsx:38` calls `buildMotionPlan` whenever `selectedDetailId`, `world`, or `baseMotionPlan` change. In `ships.ts:309-310`, `hashIdSet` short-circuits on identity (`set === prevSource`); since identity changes every render, the sort+join runs every plan rebuild.

**Change:** return `basePlan.baseEffectShipIds` directly when no `selectedShip` matches. Or memoize via `WeakMap<BasePlan, Map<string|null, Set<string>>>` keyed on `(basePlan, selectedShipId)`.

**Risk:** low — readers treat the set as readonly.

**Validation:** `npm test -- src/systems/motion.test.ts`.

### 12. Switch `cameraRef`/`canvasSizeRef` to `useLatestRef`

**Anchor:** `src/hooks/use-canvas-resize-and-camera.ts:87-92` mirrors `camera`/`canvasSize` into refs via `useEffect`. The codebase already adopted `useLatestRef` (`hooks/use-latest-ref.ts`) for `hoveredDetailIdRef`, `selectedDetailIdRef`, `motionPlanRef` (`pharosville-world.tsx:48-50`) explicitly to avoid the StrictMode desync and per-state-change extra effect.

**Change:** swap to `useLatestRef(camera)` / `useLatestRef(canvasSize)`.

**Risk:** trivial; consumers read `.current` from event handlers/RAF, never during render.

**Validation:** existing canvas-interaction tests.

### 13. Drop `world` dep from `motionPlan` memo

**Anchor:** `src/pharosville-world.tsx:38`:
```ts
const motionPlan = useMemo(
  () => buildMotionPlan(world, selectedDetailId, baseMotionPlan),
  [baseMotionPlan, selectedDetailId, world],
);
```
`buildMotionPlan` only reads `world` to find the selected ship by id. Since `motionPlan` only diverges from `baseMotionPlan` by the selected ship being added to `effectShipIds`, the `world` dep is redundant once `baseMotionPlan` identity already keys on `motionPlanSignature(world)`.

**Change:** drop the `world` dep; rely on `baseMotionPlan` identity.

**Risk:** low; tests cover signature semantics. Combine with #11.

**Validation:** `npm test -- src/systems/motion.test.ts`.

---

### 14. Logo-asset effect: depend on signature, not three array refs

**Anchor:** `src/hooks/use-asset-loading-pipeline.ts:151` deps `[assetManager, world.docks, world.graves, world.ships]`. After a refetch that produces a new `world`, all three arrays are new even when their `logoSrc` content is identical. Body recomputes `uniqueLogoSrcs`, then bails via `logoSourcesSignatureRef` comparison at line 141 — but only after building two arrays plus a `Set`.

**Change:** compute the signature at the call site (or hoist it to `useMemo`) and use the **signature string** as the effect dep.

**Risk:** trivial; signature already exists.

**Validation:** unit test that two structurally-identical worlds with different references skip the effect body.

## Phase 3 — manifest cold-load

### 15. Strip authoring-only fields from runtime manifest variant

**Anchor:** `public/pharosville/assets/manifest.json` is **50 KB** raw (was 31 KB in NFS3 baseline) — preloaded with `<link rel="preload" as="fetch">` so it sits on the critical path. `prompt`, `promptProvenance`, `semanticRole`, `criticalReason`, `paletteKeys` strings dominate. None are read by `asset-manager.ts`.

**Change:** at build time, emit `manifest.runtime.json` that strips authoring-only fields; have `PHAROSVILLE_ASSET_MANIFEST_PATH` point at the runtime variant. Keep the full manifest for the validator.

**Risk:** validator must keep targeting the full manifest; runtime tests must target the trimmed shape.

**Validation:** `npm run check:pharosville-assets` (full manifest) + asset-manager test against trimmed shape.

---

### 16. Phase split: `shellCritical` / `visibleCritical` / deferred

**Anchor:** `public/pharosville/assets/manifest.json` + `src/renderer/asset-manager.ts:131`. NFS3-04 landed budgets but not phases. `loadCritical` loads all 27 critical+firstRender assets before paint.

**Change:** introduce a `shellCritical` phase = terrain + lighthouse + main island only (target: 8-10 entries, <120 KB) so the canvas can paint a coherent first frame, then `visibleCritical` → deferred. Update validator to enforce `shellCritical` byte/decoded-pixel budgets distinctly from `visibleCritical`.

**Risk:** first paint must remain coherent — careful staging and visual review of intermediate frames.

**Validation:** `npm run check:pharosville-assets`; Playwright "desktop canvas shell" scenario; manual cold load on Fast 4G.

## Phase 5 — test coverage to lock NFS2 invariants

NFS2 #17 promised tests for two motion invariants that never landed:

### T1. Smoothstep mooring blend C0-continuity

**Anchor:** `src/systems/motion.test.ts`. Walk `progress ∈ {0, 1e-6}` and `{1-1e-6, 1}` for a moored→departing and arriving→moored transition; assert `|tile_t1 − tile_t0| < epsilon`. Catches any future regression in `applyMooringBlendInto`.

### T2. Moored ships do not draw wake even when in `effectShipIds`

**Anchor:** `src/systems/motion.test.ts`. Construct a moored sample, include the ship in `effectShipIds`, assert the `drawsWake` predicate from `ships.ts:433-435` returns false. The sample wake intensity is `0.05`/`0.03` for moored states (`motion-sampling.ts:615, 727`); the only thing keeping wake off the canvas is the state gate at the draw site — a single edit to that gate would silently regress.

Both are pure additions, ~30-50 LOC each. Land alongside Phase 1 since they protect the surfaces the renderer pass keeps touching.

## Sequencing rationale

**Phase 1** lands the highest-leverage wins with the least visual or correctness risk. #1 (Zod side-effects) is the single biggest cold-load win available; #2 (drag-pan hit-target) is the single biggest interactive-frame win still on the table; #3, #4, #5, #6, #7 each cut a discrete per-frame or critical-path source.

**Phase 2** layers on coherence and per-frame churn. Items group by file (#8/#9/#10 are gradient cache patterns; #11/#12/#13/#14 are React/effect dep cleanups). Land #11 + #13 as a single PR.

**Phase 3** is the cold-load tail. #15 is mechanically simple; #16 is the only M-effort item in the plan and benefits from #1+#3 having shrunk the JS+critical-asset critical paths first.

**Phase 4** micro-cleanups should ride along whenever the relevant file is open — none are worth a dedicated PR alone, but they each shave allocations or string churn off the hot path.

**Phase 5** locks in NFS2 invariants. Land before any further motion or wake refactor.
